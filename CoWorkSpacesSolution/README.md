Decisiones de Arquitectura y Trade-offsArquitectura Limpia (Clean Architecture)Se organizó la solución dividiendo las responsabilidades en capas desacopladas para garantizar el mantenimiento a largo plazo:Solution.Core: Contiene las entidades, interfaces y DTOs de negocio sin dependencias externas.Solution.Infrastructure: Implementa el acceso a datos mediante Dapper y centraliza transacciones usando el patrón Unit of Work.Solution.API: Expone los endpoints REST controlando las peticiones del cliente.Trade-offs (Compromisos de Diseño)Lógica en Base de Datos vs Capa de Aplicación: Se delegó el cálculo de tarifas dinámicas y el cruce de horarios a Stored Procedures. Trade-off: Se pierde portabilidad entre distintos motores SQL, pero se gana una velocidad de procesamiento masiva al evitar latencia de red trayendo registros históricos hacia la memoria de .NET.Dapper vs Entity Framework Core: Se eligió Dapper por su rendimiento crudo en lecturas y ejecuciones directas. Trade-off: Se pierde la abstracción automática de un ORM completo, requiriendo el mapeo explícito de parámetros a través de DynamicParameters.🔒 3. Justificación de la Estrategia de ConcurrenciaPara evitar el problema clásico de la Doble Reserva (dos usuarios adquiriendo el mismo espacio en el mismo bloque horario), se descartó el bloqueo optimista tradicional. En su lugar, se optó por un enfoque de Bloqueo Pesimista Estricto integrado en la base de datos mediante niveles de aislamiento avanzados.Justificación TécnicaEl Stored Procedure implementa el nivel de aislamiento SERIALIZABLE en combinación con la sugerencia de bloqueo UPDLOCK sobre las consultas de validación horaria.Cuando ingresan dos transacciones simultáneas:La primera petición lee el rango de horarios y coloca un candado de actualización (UPDLOCK) que se mantiene activo hasta que termine toda su transacción.La segunda petición, al intentar leer el mismo espacio, es forzada a esperar en hilera ("fila india").Una vez insertada la primera reserva, la segunda despierta, detecta el cruce inmediato y aborta de forma segura mediante un RAISERROR('CONFLICT_409'). Esto garantiza consistencia absoluta de datos a nivel ACID.📊 4.
